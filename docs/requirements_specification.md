# 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名

**「ひきこもり支援 × AI 最適化プラットフォーム開発プロジェクト」**

### 1.2 背景・目的

**背景**
- 国内のひきこもり人口は約146万人、経済損失は11兆円規模と推計されている
- 家族内支援にとどまるケースが約70%を占め、医療・福祉支援に繋がらない現状がある
- 支援者は「予想」や「経験」に依存し、自己開示が少ない当事者には適切な支援が難しい

**目的**
- 会話データとバイタルデータを掛け合わせて「支援の最適化」を実現する
- 熟練支援者の経験に頼らず、データドリブンな支援プロセスを提供する
- ひきこもり当事者の社会復帰までの期間短縮と支援継続率の向上を目指す
- **競合優位性**: 客観的バイタルデータ評価による主観評価の限界を克服

### 1.3 システムのビジョン / スコープ

**ビジョン**
- データドリブンな支援により、ひきこもり当事者一人ひとりに最適化された支援を提供し、社会復帰への道筋を明確化する
- 支援者・家族・当事者すべてが安心して利用できる、科学的根拠に基づく支援プラットフォームを構築する
- 主観評価の限界を克服し、客観的バイタルデータによる支援の質向上を実現する

**スコープ**
- 今回はWebアプリケーションとしてMVPを実装
- 会話データ収集・分析機能、バイタルデータ収集・分析機能、支援者向けダッシュボード機能をコア機能として実装
- 将来的なスマートフォンアプリ展開やハードウェア連携は本フェーズでは対象外
- 自治体連携機能は基本的なデータ連携に限定

## 2. ビジネス要件

### 2.1 ビジネスモデル情報

**リーンキャンバス要約**
- **解決する課題**: ひきこもり支援における経験依存型アプローチの限界、客観的指標の不足
- **価値提案**: 会話×バイタルデータによる科学的根拠に基づく支援最適化
- **収益構造**: 
  - 初期: 自治体助成金（年間予算からの請求書払い）
  - 将来: 家族直接支払い（3ヶ月に1回5,000円）
- **顧客セグメント**: 
  - 初期: 自治体（助成金による導入）
  - 将来: ひきこもり当事者の家族（直接契約）

**7 Powers分析**
1. **規模の経済**: データ量増加によりAI解析精度向上、1ユーザーあたりコスト逓減
2. **ネットワーク経済**: 施設増加により支援精度向上、支援者・当事者双方の価値向上
3. **カウンターポジショニング**: 既存支援モデル（経験・勘頼み）とは異なるアプローチ
4. **スイッチングコスト**: コミュニティやノウハウ共有を失うリスクによる離脱防止
5. **ブランド力**: 安心感重視の「寄り添い型」メッセージによる信頼構築
6. **囲い込んだリソース**: 独自バイタルセンサー、実証事業データ、支援施設とのPoC
7. **プロセス力**: 自治体や支援施設とのPoC積み重ねによる独自連携プロセス

**競合分析**
- **メンタルヘルス定量評価**: 
  - 既存: うつの質問表（主観評価、ブレが生じる）
  - 当社: 客観的バイタルデータ評価（会話×バイタル×支援事例）
  - **差別化要因**: 客観評価を進めるのは当社のみ
- **支援アプローチ**: 
  - 既存: VRを用いたアプローチ（検討段階）
  - 当社: データドリブンな支援最適化
  - **差別化要因**: 科学的根拠に基づく個別最適化

**市場規模・成長予測**
- 国内ひきこもり人口: 約146万人
- 経済損失: 11兆円規模
- ターゲット市場: 支援機関数（全国約1,000施設）
- **初年度目標**: 2施設での導入開始
- **成長計画**: 200%の事業成長（年次）
  - 2年目: 4施設
  - 3年目: 8施設
  - 4年目: 16施設
  - 5年目: 32施設
- **中期目標**: 5年以内に32施設での導入、年間売上3.2億円達成
- **海外展開計画**: 
  - 時期: 5年目以降
  - 前提条件: 日本でエビデンスを明確化
  - 対象国: アメリカ等（セルフモニタリングが充実した市場）

- **社会復帰までの期間短縮**: 平均2年短縮（現在の推定5年から3年に短縮）
- **利用施設数**: 
  - 初年度: 2施設での導入
  - 2年目: 4施設（200%成長）
  - 3年目: 8施設（200%成長）
- **利用件数**:

  - 2年目: 約120件
  - 3年目: 約240件
- **収益目標**: 
  - 初年度: 自治体助成金による収益（年間予算からの請求書払い）
  - 将来: 家族直接支払い（3ヶ月に1回5,000円×利用件数）
- **顧客満足度**: 支援者満足度80%以上、家族満足度75%以上

**追跡方法**

- **月次**: 定着率、契約数、利用件数のモニタリング
- **四半期**: アルゴリズム改善サイクル実施、KPI見直し
- **年次**: 信頼性向上のための論文執筆・発表、自治体・省庁との連携強化
- **5年目以降**: 海外展開準備（アメリカ等への市場調査、現地パートナーシップ構築）

**ビジネスモデル健全性指標**
- **LTV/CAC比率**:

  - 将来モデル: 0.5〜1.6倍（要改善）
- **CAC回収期間**: 
  - 目標値: 12ヶ月以内
  - 初期モデル: 3〜6ヶ月（優秀）

  - **将来モデルの収益性改善**:

    - 施設規模拡大: 1施設あたり当事者数を20〜30人に増加
    - プレミアム機能: 個別カウンセリング、家族向けサポート追加
  - **CAC削減**: デジタルマーケティング活用、口コミ・紹介制度導入
  - **収益最適化**: 段階的料金体系、追加サービス提供

### 2.3 ビジネス上の制約

**予算・開発期間・リソース制約**
- **開発予算**: 年間1,000〜1,500万円（人件費含む）
- **システム運用費**: 年間200〜300万円（クラウド/AI基盤）
- **営業コスト**: 1施設あたり45〜75万円（CAC）
- **開発期間**: MVP開発6か月、本格運用開始まで12か月
- **収益化モデル**: 
  - 初期: 自治体助成金（年間予算からの請求書払い）
  - 将来: 家族直接支払い（3ヶ月に1回5,000円）

**CAC・LTV分析**
- **CAC（Customer Acquisition Cost）**: 1施設あたり45〜75万円
- **LTV（Lifetime Value）計算**:
  - **初期モデル（自治体助成金）**: 
    - 年間助成金額: 約200〜300万円/施設
    - 契約期間: 3〜5年想定
    - LTV: 600〜1,500万円/施設
  - **将来モデル（家族直接支払い）**:
    - 1施設あたり当事者数: 10〜20人想定
    - 1人あたり年間収益: 20,000円（5,000円×4回/年）
    - 1施設あたり年間収益: 20〜40万円/施設
    - 利用期間: 2〜3年想定（社会復帰まで）
    - LTV: 40〜120万円/施設（20〜40万円×2〜3年）
- **LTV/CAC比率**:
  - 初期モデル: 8〜20倍（健全）
  - 将来モデル: 0.5〜1.6倍（要改善）
- **回収期間**: 
  - 初期モデル: 3〜6ヶ月（優秀）
  - 将来モデル: 1〜2年（要改善）

**法的要件・規制**
- 個人情報保護法準拠
- **データ取り扱い方針**: 
  - 医療データでの検証は実施するが、実際の取り扱いデータは医療データではない
  - 個人情報は匿名の統計データとして使用
  - 個人に情報が紐づいたデータは使用しない
- **自治体連携**: 
  - 自治体は福祉施設との繋ぎ役のみ
  - 具体的なデータ連携は不要
  - 所管部門の承認後、助成金申請の合意済み
- 当事者家族への経済的負担軽減の考慮

## 3. ユーザー要件

### 3.1 ユーザープロファイル / ペルソナ

**主要ペルソナ**

**ペルソナ1: 支援者（精神科医・支援員）**
- 年齢: 30-50歳
- 課題: 経験に依存せず、客観的な支援指標を求めている
- 利用シーン: 日々の支援活動での意思決定支援
- 利用デバイス: PC、タブレット

**ペルソナ2: 当事者家族**
- 年齢: 50-70歳
- 課題: 経済的・精神的負担を抱えつつ支援を模索
- 利用シーン: 支援機関での相談時、将来的には直接契約
- 利用デバイス: PC、スマートフォン
- 支払い: 将来は3ヶ月に1回5,000円の直接支払い

**ペルソナ3: 福祉施設職員**
- 年齢: 30-50歳
- 課題: 既存データを活用した効果的な支援方法の模索
- 利用シーン: データインポートによる支援方法提案の受領
- 利用デバイス: PC

**ペルソナ4: 自治体担当者**
- 年齢: 30-50歳
- 課題: 孤独孤立支援を重点施策として推進中
- 利用シーン: 福祉施設との繋ぎ役、助成金申請支援
- 利用デバイス: PC
- 支払い: 初期は年間予算からの請求書払い

### 3.2 ユーザーストーリー

1. **支援者として、当事者の状態を客観的に把握したい。なぜなら経験や勘に頼らず、科学的根拠に基づいた支援を提供できるからだ。**

2. **当事者家族として、支援の進捗を可視化された形で確認したい。なぜなら将来への希望と安心感を得られるからだ。**

3. **支援者として、個別最適化された支援アプローチを提案されたい。なぜなら支援効果を最大化し、当事者の社会復帰を早められるからだ。**

4. **自治体担当者として、地域のひきこもり支援の効果を定量的に把握したい。なぜなら政策立案と予算配分の根拠とできるからだ。**

5. **福祉施設職員として、既存のデータを活用して効果的な支援方法を知りたい。なぜなら統計的根拠に基づいた支援を提供できるからだ。**

6. **自治体担当者として、福祉施設との連携を支援したい。なぜなら地域課題の解決と助成金申請の支援ができるからだ。**

7. **当事者家族として、将来的には直接契約で支援を受けたい。なぜなら3ヶ月に1回5,000円の負担で効果的な支援を受けられるからだ。**

### 3.3 MVP（Minimum Viable Product）の定義

**MVPで実装する範囲**
- 会話データ収集・分析機能（テキスト化、自然言語解析）
- バイタルデータ収集・分析機能（心拍・自律神経解析）
- 支援者向けダッシュボード（当事者の状態・推奨アプローチ表示）
- 基本的なデータ統合機能（会話×バイタル×支援事例）
- データインポート機能（福祉施設からのデータ取り込み）

**MVPのゴール**
- 早期リリースによる実証データ収集
- 支援者・当事者からのフィードバック獲得
- アルゴリズム精度の向上基盤構築
- 市場検証とビジネスモデルの検証

### 4.2 機能詳細仕様

#### 4.2.1 F-001: 会話データ収集機能

- **概要**: 支援者と当事者の会話を音声からテキストに変換し、自然言語解析を行う
- **ユースケース**: 「支援者が当事者との会話を記録し、感情・心理状態を分析したいとき」
- **前提条件**: 
  - 音声認識SaaS（Azure Speech Services等）が導入されている
  - 自然言語処理API（OpenAI GPT-4等）が利用可能
- **正常系フロー**:
  1. 支援者が会話録音を開始
  2. 音声データをリアルタイムでテキスト化
  3. テキストデータを自然言語解析APIに送信
  4. 感情分析・キーワード抽出・心理状態推定を実行
  5. 分析結果をデータベースに保存
- **例外系フロー**:
  - 音声認識精度が低い場合（雑音、方言等）
  - ネットワーク接続エラー時のオフライン処理
  - 個人情報を含む発言の自動マスキング
- **UI要件**:
  - 録音開始/停止ボタンの明確な表示
  - リアルタイムでのテキスト化結果表示
  - 分析結果の視覚的表示（感情グラフ等）
- **非機能面注意**:
  - セキュリティ（音声データの暗号化、HTTPS通信）
  - レスポンス性能（音声認識3秒以内、分析結果表示5秒以内）
  - プライバシー保護（会話内容の匿名化処理、個人情報の統計データ化）

#### 4.2.2 F-002: バイタルデータ収集機能

- **概要**: Fitbitから心拍・自律神経データを収集し、ストレス・心理状態を分析する
- **ユースケース**: 「当事者の生理的状態を客観的に把握し、心理状態の変化を検知したいとき」
- **前提条件**:
  - Fitbitデバイス（当事者に配布済み）
  - Fitbit Web API連携が設定済み
  - 脈拍データのフーリエ変換による自律神経解析システムが構築済み
- **正常系フロー**:
  1. Fitbitから脈拍データ・加速度データを自動収集
  2. Fitbit Web API経由でデータを取得
  3. 脈拍データのフーリエ変換による自律神経解析を実行
  4. 加速度データから生活リズム（活動パターン）を推測
  5. ストレスレベル・心理状態・生活リズムを数値化
  6. 分析結果をデータベースに保存
- **例外系フロー**:
  - Fitbit接続不良時のデータ欠損処理
  - 異常値検知時のアラート機能
  - デバイス充電切れ時の代替手段
  - API制限時のデータ取得遅延対応
- **UI要件**:
  - バイタルデータのリアルタイム表示
  - ストレスレベル・心理状態・生活リズムの視覚的表示
  - 自律神経バランス（交感神経・副交感神経）のグラフ表示
  - 異常値検知時のアラート表示
- **非機能面注意**:
  - セキュリティ（データの暗号化、Fitbit API認証）
  - 可用性（Fitbit API制限・障害時の継続運用）
  - 精度（Fitbitの測定精度：心拍数±3bpm、加速度±0.1g）
  - データ同期（Fitbit API制限：1日150回リクエスト）
  - プライバシー保護（バイタルデータの匿名化、個人紐づきデータの非使用）

#### 4.2.3 F-003: データ統合・分析機能

- **概要**: 会話データ・バイタルデータ・支援事例を統合し、AIによる個別最適化分析を実行する
- **ユースケース**: 「複数のデータソースを統合し、当事者に最適な支援アプローチを提案したいとき」
- **前提条件**:
  - 機械学習プラットフォーム（AWS SageMaker等）が利用可能
  - 支援事例データベースが構築済み
- **正常系フロー**:
  1. 会話データ・バイタルデータ・支援事例を統合
  2. 機械学習モデルによるパターン分析を実行
  3. 当事者の状態・特性を数値化・分類
  4. 類似事例との比較分析を実行
  5. 最適な支援アプローチを提案
- **例外系フロー**:
  - データ不足時の分析精度低下対応
  - モデル更新時の分析結果変更対応
  - プライバシー保護のためのデータ匿名化（個人紐づきデータの非使用）
- **UI要件**:
  - 分析結果の統合ダッシュボード表示
  - 支援アプローチ提案の詳細表示
  - 類似事例の比較表示
- **非機能面注意**:
  - パフォーマンス（分析処理10秒以内完了）
  - セキュリティ（データの匿名化、アクセスログ）
  - **信頼性・精度**:
    - 現時点: 定性的信頼性（自治体・省庁支援による）
    - 将来目標: 定量的信頼性（データ活用支援論文の執筆）
  - プライバシー保護（統計データ化、個人紐づきデータの非使用）

#### 4.2.4 F-004: 支援者ダッシュボード

- **概要**: 当事者の状態・推奨アプローチ・支援進捗を統合的に表示する支援者向けインターフェース
- **ユースケース**: 「支援者が当事者の状況を把握し、効果的な支援を計画・実行したいとき」
- **前提条件**:
  - データ統合・分析機能が正常に動作している
  - 支援者アカウントが作成済み
- **正常系フロー**:
  1. 支援者がダッシュボードにログイン
  2. 担当当事者の一覧を表示
  3. 選択した当事者の詳細情報を表示
  4. 推奨アプローチ・注意点を確認
  5. 支援計画の更新・記録
- **例外系フロー**:
  - データ更新遅延時の表示対応
  - 複数支援者による同時アクセス対応
  - 緊急時のアラート表示
- **UI要件**:
  - 直感的なナビゲーション設計
  - 重要な情報の優先表示
  - レスポンシブデザイン（PC・タブレット対応）
- **非機能面注意**:
  - ユーザビリティ（操作習得時間30分以内）
  - セキュリティ（ロールベースアクセス制御）
  - 可用性（24時間365日アクセス可能）

#### 4.2.5 F-005: データインポート機能

- **概要**: 福祉施設から会話テキスト・バイタルデータをインポートし、統計的データに基づいた支援方法を提案する
- **ユースケース**: 「福祉施設が既存のデータを活用して、統計的根拠に基づいた支援方法を知りたいとき」
- **前提条件**:
  - 福祉施設のデータ形式（CSV、Excel等）が確定している
  - データマッピング機能が実装済み
- **正常系フロー**:
  1. 福祉施設がWebアプリケーションにログイン
  2. データインポート画面でファイルをアップロード
  3. データ形式の自動検証・マッピング
  4. 統計的データに基づいた支援方法を提案
  5. 提案結果をダッシュボードに表示
- **例外系フロー**:
  - データ形式エラー時の修正指示
  - 個人情報を含むデータの自動検知・除外
  - データ量制限超過時の分割処理
- **UI要件**:
  - ドラッグ&ドロップでのファイルアップロード
  - データプレビュー機能
  - インポート進捗の表示
- **非機能面注意**:
  - セキュリティ（ファイルアップロードの検証、マルウェア対策）
  - パフォーマンス（大容量ファイルの処理）
  - プライバシー保護（個人情報の自動検知・除外）

#### 4.2.6 F-006: ユーザー認証機能

- **概要**: 支援者・管理者のログイン・権限管理を行う認証システム
- **ユースケース**: 「支援者が安全にシステムにアクセスし、適切な権限で機能を利用したいとき」
- **前提条件**:
  - 認証SaaS（Auth0、Firebase Auth等）が導入されている
  - 組織・役割管理システムが構築済み
- **正常系フロー**:
  1. ユーザーがログイン画面で認証情報を入力
  2. 認証SaaSによる認証処理
  3. JWTトークンによるセッション管理
  4. ロールベースの権限確認
  5. 適切なダッシュボード画面に遷移
- **例外系フロー**:
  - 認証失敗時のエラーメッセージ表示
  - セッションタイムアウト時の再認証
  - 権限不足時のアクセス制限
- **UI要件**:
  - シンプルなログイン画面
  - パスワードリセット機能
  - 多要素認証対応
- **非機能面注意**:
  - セキュリティ（JWT暗号化、CSRF/XSS対策）
  - 可用性（認証システムの高可用性）
  - パフォーマンス（認証処理2秒以内）

## 5. 非機能要件

### 5.1 パフォーマンス要件

- **レスポンス時間**: 
  - 主要画面表示: 3秒以内
  - データ分析結果表示: 10秒以内
  - 音声認識処理: 3秒以内
- **同時接続数**: 
  - 初年度: 20ユーザー同時アクセスに対応（2施設×10ユーザー想定）
  - 2年目: 40ユーザー同時アクセスに対応
  - 3年目: 80ユーザー同時アクセスに対応
- **処理量**: 
  - 初年度: 1日あたり約20リクエスト（2施設×10当事者想定）
  - 2年目: 1日あたり約40リクエスト
  - 3年目: 1日あたり約80リクエスト
- **データ処理**: リアルタイムバイタルデータ処理（1秒以内の遅延）

### 5.2 セキュリティ要件

- **認証/認可**: 
  - Auth0等の外部認証サービス利用
  - JWT（JSON Web Token）によるセッション管理
  - ロールベースアクセス制御（RBAC）
- **データ保護**: 
  - HTTPS通信の強制
  - 個人情報・医療データの暗号化保存（AES-256）
  - データベース暗号化
- **監査ログ**: 
  - 管理者による不正アクセス/操作ログ記録
  - データアクセス履歴の保持（3年間）
  - セキュリティイベントの自動検知
- **コンプライアンス**: 
  - 個人情報保護法準拠
  - **データ匿名化**: 個人情報は匿名統計データとして使用、個人紐づきデータは使用しない
  - **医療データ規制**: 検証用途でのみ使用、実際の取り扱いデータは医療データではない
  - GDPR対応（将来的な海外展開時）

### 5.3 可用性・信頼性

- **稼働率**: 99.5%以上（月間ダウンタイム36時間以内）
- **障害時の復旧手順/DR対応**: 
  - 日次バックアップ（データベース）
  - 週次フルバックアップ
  - 復旧時間目標（RTO）: 4時間以内
  - データ復旧目標（RPO）: 1時間以内
- **フェイルオーバー**: 
  - マルチAZ構成による冗長化
  - 自動フェイルオーバー機能
  - 負荷分散による継続運用

### 5.4 ユーザビリティ / UI・UX

- **アクセシビリティ**: 
  - WCAG 2.1 AA準拠
  - スクリーンリーダー対応
  - キーボードナビゲーション対応
- **多言語対応**: 
  - 主に日本語対応
  - **海外展開対応**: 5年目以降のアメリカ等展開に向けた英語対応を考慮した設計
- **操作導線**: 
  - 3クリック以内での主要機能アクセス
  - 直感的なアイコン・ラベル使用
  - エラー時の分かりやすいメッセージ表示

### 5.5 スケーラビリティ

- **水平/垂直スケーリング**: 
  - クラウド環境（AWS）のオートスケール機能活用
  - コンテナ化（Docker）による柔軟なスケーリング
  - マイクロサービスアーキテクチャ採用
- **突発的アクセス増対策**: 
  - CDN（CloudFront）による静的コンテンツ配信
  - キャッシュ戦略（Redis）による応答速度向上
  - リトライ制御・サーキットブレーカーパターン

  - ~~医療データ規制の変更~~（検証用途のみ、実際の取り扱いデータは医療データではない）
  - Fitbit API制限・サービス変更リスク
  - **信頼性・精度の課題**:
    - 現時点での定量的信頼性の不足
    - 論文執筆・発表による信頼性向上の時間的制約
  - フーリエ変換による自律神経解析の精度検証
  - データ匿名化の技術的実装

    - VRを用いた支援アプローチの技術進歩

    - 現地規制対応（HIPAA等の医療データ規制）
    - 文化・言語の違いによる市場適応
    - 現地競合他社の存在
